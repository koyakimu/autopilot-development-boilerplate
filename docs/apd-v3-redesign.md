# APD v3 再設計

> Status: 設計（自己レビュー反映済み・実装計画前）
> Date: 2026-06-28
> バージョン: 2.0.0 → 3.0.0（互換性なしの変更を含む）

## 1. なぜ作り直すか

Claude Code 本体（`/goal`, hooks, subagent など）と公式プラグイン（superpowers, pr-review-toolkit
など）が充実し、「開発を自動で速く回す道具」は出揃った。そのうえで APD には2つの課題がある。

1. **役割が被って見える** — superpowers のフェーズと APD のフェーズが似ていて、APD の独自性が分かりにくい。
2. **使いこなせない** — 道具が多すぎて「いつどのコマンドを打てばいいか」分からない。

方針: **公式にある機能は全部任せ、APD は独自の部分だけに絞る。**

### ユーザーが一番困っていること（出発点）

> 時間をかけて Spec を作ったのに、**完成する前に確認を求められて作業が止まる**。

だから消すべきは「人間の確認」ではなく「**完成前の中断**」。確認は上流（意図・Spec）と
完成後（実機確認）にまとめ、**実装の最中は止めない**。

## 2. APD の立ち位置

> **公式の道具は「作業」を担う。APD は「段取り」を担う。**
> 作業が自動化されるほど、人が関わる数少ない場面（どこで判断し、どこまで任せるか）の
> 設計が効いてくる、という賭け。

「作業は公式に任せ、段取りだけ APD が持つ」が背骨。

### 公式を調べて分かったこと

- superpowers の Spec・計画ファイルは**日付つきの使い捨て**で溜まっていく。APD の「1枚を直し続ける」とは逆。
- コードレビューの**実行は AI が完結**できる（人がコードを読まなくてよい）→ 任せられる。
- 「Spec どおりに作れているか」のチェックは公式にも一部あるが、**その場限り・1回だけ**。
  **プロダクトの軸（Design）を基準に、実装中ずっと続けるチェックは無い** → ここが APD の独自領域。
- 使える仕組みを発見: Claude Code の **`type:"agent"` の Stop フック**。AI が止まろうとするたびに
  自動で走り、design.md や Spec や差分を**直接読めて**、ダメなら**止めて続行させられる**。
  これを Spec チェックの土台にする（新しめの機能なので、実装前に動作確認する）。

## 3. 一番の変更 — 実装中は止めない

「フェーズ（作業の段階）」と「人が関わる場面」を分けて考える。減らすのは
**完成前に止まる回数（理想ゼロ）**であって、作業の段階ではない（チェックはむしろ増やす）。

| | 旧 v2 | 新 v3 |
|---|---|---|
| 人が関わる場面 | 意図 + **実装中の確認** + 完成後の実機確認 | 意図 + Spec 承認 + 完成後の実機確認 |
| 実装中に止まるか | 止まる（重要な判断で人に確認） | **止まらない（最後まで自動で走る）** |
| 完成後の実機確認 | あり（OK でマージ） | **あり（止めない／違えば次サイクルで直す）** |

> v2 と v3 の差は**「実装中の確認をなくし、実装を止まらない自動ブロックにする」**だけ。
> 完成後の実機確認は**なくさない**。「マージの関門」から「見て、違えば次でやり直す確認」に変える。

### 用語の整理

- **実機確認** = 完成後に人が実際に動かして見る。v3 でも残る。
- **実装中の確認** = 作業途中で人に判断を仰いで止まること。v3 で**なくす**。
- **受け入れ条件（AC）** = Spec に書く「これを満たせば OK」の条件。v3 で**残す・むしろ主役**（チェックの基準）。

### 新しい流れ

```
① 意図を伝える                       ← 人（上流）
② Spec を確認して承認する             ← 人（上流・最後の事前確認）
        ↓
┌────────────────────────────────┐
│  AI が最後まで自動で走る（途中で止まらない）        │
│  実装 → テスト → /security-review → Spec チェック │
│       → 出せる状態の PR                        │
└────────────────────────────────┘
        ↓
③ 完成後の実機確認  ← 人が見る（止めない・違えば Spec を直して次サイクル）
```

- 違うものが出たら、Spec を直してもう一度 `/apd:go`（戻すのではなく次に進む）。
- 違いに気づく役は**③の実機確認（＝開発者本人）**。これで「Spec の中だけで答え合わせして
  間違いに誰も気づかない」状態を避ける（§10）。

## 4. 公式に任せるもの（APD は持たない）

| やること | 任せ先 |
|---|---|
| 実装ループ | `/goal`（条件を満たすまで継続）＋ Spec チェックの Stop フックが番人兼エンジン |
| Spec チェックの**実行** | 本体の `type:"agent"` Stop フック（毎回・ファイル直読・止められる） |
| 並列・サブエージェント・隔離 | superpowers の並列スキル、本体の agent teams、Agent の worktree 隔離 |
| TDD・デバッグ・完了確認 | superpowers の各スキル |
| 一般コードレビュー（バグ・型・テスト・握り潰し・コメント） | pr-review-toolkit、code-review プラグイン |
| セキュリティレビュー | `/security-review`（実装中に自動実行、結果をチェックに反映） |
| todo・巻き戻し・ブランチ統合 | Task系 / `/rewind` / superpowers の統合スキル |

## 5. APD が独自に持つもの（3つ）

1. **プロダクトの軸（Design）と、直し続ける Spec** — 既存の `apd:design` / `apd:spec`。
2. **Spec チェック（心臓）** — 次節。最大の独自性。ただし**唯一の番人ではない**（③実機確認も残る）。
   独自なのは「**プロダクトの軸を基準に、実装中ずっと続ける**」点で、実行の仕組みは公式の Stop フックに乗る。
3. **人が関わる場面を上流2点＋完成後1点に絞り、実装中をゼロにする段取り** —
   ユーザーの「使い分けが分からない」「途中で止まる」を直接解く。

## 6. Spec チェックの設計

| 項目 | 内容 |
|---|---|
| 仕組み | 本体の `type:"agent"` Stop フックとして APD プラグインに登録 |
| 基準 | 承認済みの **AC + Design** |
| 入力 | Spec・Design・実装差分（フックが自分で読む） |
| 種類 | **差分・ファイルの照合（コードは動かさない）**。実際の動作はテストと CI が見る |
| 出力 | 条件ごとに「OK / ずれ / 未実装」＋ 根拠（ファイル:行）＋ **保証範囲の明示**。PR コメント等に残し、後から人が監査できるようにする |
| タイミング | **毎回（AI が止まるたび）**。フックなので確実に走る（AI の自己申告に頼らない） |
| ダメなとき | 止めて AI に続行させ、自動で直す（人に確認を求めない＝完全自動） |
| Spec バグの疑い | ずれが「実装ミス」か「Spec の誤り」かをチェックが判定。Spec バグでも**実装は止めず**、その旨を出力に残して③に送る |

### spike 結果（2026-06-28・実機確認済み）

`type:"agent"` Stop フックを実機検証し、本設計の前提が成立することを確認した:

- ✅ フックが発火し（`matcher: ""` で動作）、`ok:false`＋reason で実装を続行させられる
- ✅ フック内のエージェントが **Read ツールでリポジトリのファイルを直読できた**（design.md / Spec / 差分を基準に検証できる）

結論: subagent フォールバックは不要。type:agent Stop フックで直読・検証する本設計で進める。

### `/goal` の判定との関係

`/goal` の合否判定は「会話に出てきた情報」だけで決まる（APD の現行 start skill の前提。本体の
公式ドキュメントでの確認は未了＝**今の観察に基づく前提**）。なので達成条件に
「Spec チェックの最新結果が OK」を含める。**実体のチェックはフックが自分で行う**ので、
AI が「できた」と言うだけでは通らない。

### 非機能要件（性能・セキュリティなど）

- 数値で測れるもの（性能の閾値、a11y など）は **AC に書いてチェック対象**にする。
- 書けない/任せるもの（セキュリティ等）は**任せ先（`/security-review` 等）の結果を達成条件に含める**。

### 詰まったときの逃げ道

Spec の矛盾などでチェックが永遠に通らない場合に備える: `/goal` の上限に任せつつ、
通らない状態が続いたら**途中までを下書き PR で残して止め**、③の実機確認に回す。

## 7. コマンド（人が打つ数を最小に）

```
/apd:design   プロダクトの軸を作る（最初と大きな方針転換のときだけ）
/apd:spec     意図 → Spec を作って承認（上流の確認）
/apd:go       Spec から達成条件を組み立てる（旧 /apd:start を改名・強化）
```

**正直な制約**: コマンドはユーザー入力からしか起動しないので、`/apd:go` が中で勝手に `/goal` を
起動することはできない。`/apd:go` は**達成条件を作って提示し、ユーザーがそれを `/goal` に貼って起動**する。
実際に打つのは「design（稀）＋ spec ＋ go ＋ /goal 貼付」。起動後は Spec チェックのフックが番人兼エンジンになり、
**実装中は止まらず完走**する。`/apd:go` の出力はそのまま `/goal` に貼れる形にする。

## 8. サジェスト機能（2種類）

「いつどのコマンドを打つか分からない」を hooks で助ける。Spec チェック用のフック（§6）とは別。

- **状態サジェスト（止めない）** — SessionStart / Stop フックで、`docs/apd/` のファイルや
  `gh pr list` を見て次のコマンドを**提案するだけ**（実行はしない）:
  - design.md が無い → `/apd:design`
  - design.md ができた → `/apd:spec`
  - Spec を承認した → `/apd:go`（→ 出力を `/goal` に貼る案内まで）
  - `/apd:go` が完走した → ③実機確認を案内
- **Spec チェック（止める）** — §6。実装中の番人。
- **原則: 出しすぎない**。状態サジェストは切り替わりの節目だけ。
- 設定は `update-config` スキルで正確に組む。

## 10. リスクと、その受け止め

- **Spec の誤りに気づけない**: チェックは「実装 ⇄ Spec」しか見ないので、Spec 自体が間違っていると
  OK のまま間違った方向に完走しうる。→ **③実機確認**で気づき、Spec を直して次サイクル。
  チェックの出力に「Spec 自体の正しさは保証しない」と明記する。
- **OK 表示を過信する**: 「チェック OK」を全体品質の保証と読ませない。出力に保証範囲
  （Spec との一致だけ・動作やセキュリティ・使い心地は別）を明記する。
- **セキュリティ等の実機リスク**: v2 は人に確認していたが、v3 は止めない方針。`/security-review`（静的）
  ＋③実機確認で見る。実機でしか出ない問題が OK 表示で完走しうる残りリスクは、
  **完全自動を優先するユーザーの判断として受け入れる**（違えば次サイクルで直す）。
- **Spec に書けない品質**: 使い心地・見た目・実機限定の挙動 → ③実機確認が担う。
- **チェックの負荷**: 毎回チェックのコスト（差分が小さい回は軽くする等を実装時に検討）。
- **`type:agent` の成熟度**: 新しめの機能。実装前に動作確認し、ダメなら代替（AI が照合して結果を会話に出す）を残す。
