# Harness Engineering 取り込み調査

Anthropic「Harness design for long-running apps」のAPDフレームワークへの取り込み検討（2026-04-29実施）

## 背景

APDフレームワークは Spec-Driven Development を軸に設計してきたが、Build Phase（AI自律実行）の長時間化、評価ループの収束性、品質スコアリングなど、ハーネス設計の観点ではまだ規定が弱い領域がある。Anthropic公式の事例記事を起点に、APDで採用すべき要素・採用しない要素を整理する。

**Why:** AIエージェントが長時間自律実行する以上、ハーネス（外側の制御構造）の設計品質が成果物の品質を直接決める。記事の知見は AnthropicがClaude本体で実証した実践知であり、APDの汎用フレームワークに取り込む価値が高い。

**結論:** 既存のマルチエージェント＋Checkpoint構造はおおむね整合した。議論の結果、**コンテキストリセット規約（A-1）**、**評価ループの収束基準（A-2）**、**動作検証と3点突合（B-2）** をフレームワーク本体に取り込んだ。**rubric評価（B-1）**は A-2 / B-2 の「Specに評価軸を書く」設計に統合された。**コスト/時間制約（C-1）** は規定しない方針。**ハーネスの陳腐化マネジメント（C-2）** は APD-FRAMEWORK.md 末尾に節として追加した。

## 記事の要点

### 1. ハーネスの定義

複数のエージェントを役割分担（Planner / Generator / Evaluator）させ、長時間タスクで一貫した品質を保つための構造化システム。単一モデル呼び出しでは到達できない品質域を狙う。

### 2. 主要原則

**コンテキスト管理**
- 長時間タスクではモデルの一貫性が崩れる
- 「コンテキストリセット」 — 新セッション + 状態引き継ぎファイルでクリーンスレートを与える
- 単純な圧縮より有効

**タスク分解**
- スプリント形式で機能単位に分割
- Plannerが曖昧プロンプトを詳細仕様に展開

**自己評価の分離**
- モデルは自作物を肯定的に評価する傾向がある
- 独立Evaluatorで客観性を担保

### 3. 構成要素

| 構成要素 | 役割 |
|---|---|
| Planner | 曖昧要件 → 詳細仕様 |
| Generator | 実装（スプリント単位） |
| Evaluator | Playwright等で実動作確認、バグ検出 |
| スプリント契約 | 実装前の「完成の定義」合意 |
| 構造化ファイル | エージェント間のコンテキスト引き継ぎ |

**評価基準（rubric）の例**: フロントエンドでは「デザイン品質 / オリジナリティ / クラフト / 機能性」を多軸スコア化することで主観評価を測定可能にする。

### 4. アンチパターン

- 自己評価への依存（過度に肯定的になる）
- 表面的なテスト（エッジケース未検証）
- 過度な仕様化（Plannerが実装詳細まで指定すると誤りが下流に波及）
- コンテキスト圧縮への過信（クリーンスレートのほうが有効）

### 5. 実装テクニック

- フロントエンド設計ループ: Generator初回出力 → EvaluatorがPlaywrightで実ナビゲーション+スクリーンショット確認 → 詳細批評 → 5〜15回反復でスコア改善まで継続
- DAW製造例: Opus 4.6では「スプリント構造廃止 + 単一ビルド」で2時間以上の一貫動作を実現
- Evaluatorは初期段階で甘い判定になりがち。ログレビューと段階的なプロンプト改善が必須
- コスト/時間トレードオフ: 単一エージェント20分・$9 → フルハーネス6時間・$200（20倍コストで大幅品質向上）→ 改定版（スプリント廃止）3.8時間・$125

### 6. モデル進化への適応

新世代モデルでは計画/長文/コード審査能力が向上し、以前は必須だったハーネス要素が不要になる。**ハーネスは継続的に簡素化を再検討すべき**。

## APDとの対応マッピング

| 記事の要素 | APDの現状 | ギャップ |
|---|---|---|
| Planner / Generator / Evaluator の役割分離 | リーダーエージェント / 実装エージェント / Peer Review + 専任Checkpoint で対応済 | ほぼなし |
| 自己評価バイアス回避（独立Evaluator） | Peer Review（クロスコンテキスト）+ 専任Checkpoint で分離済 | ほぼなし |
| タスク分解 | サイクル型 + Phase 1のスコーピングで対応 | ほぼなし |
| **コンテキストリセット**（新セッション + 引き継ぎファイル） | Phase間でドキュメント引き継ぎはする。Build Phase **内**での明示的なリセット規定はなし | **未対応** |
| **イテレーティブ評価ループ**（収束まで5〜15回） | `request_changes` で戻す仕組みはあるが、回数・収束基準が暗黙的 | **規定なし** |
| **採点基準（rubric）** | Exit Criteria は基本「満たす/満たさない」の二値。多軸スコアの仕組みなし | **未対応** |
| **実動作検証**（Evaluatorが実際に触る） | Human Checkpoint 2 で人間が確認。AIによる自動動作検証は個別実装任せ | **規定なし** |
| **ハーネスのコスト/時間制約** | タイムボックス・費用上限の概念なし | **未対応** |
| **ハーネスの陳腐化マネジメント** | モデル進化に応じた簡素化の規定なし | **未対応** |
| 過度な仕様化の回避 | Phase 1の責務はSpecまで。実装詳細はBuild内のリーダーが計画する設計 | 整合 |

## 取り込み候補（優先度つき）

### A. 汎用的に効く（取り込み推奨）

**A-1. コンテキストリセット規約**
- 長時間Buildで、リーダーエージェントが状態引き継ぎファイル（handoff document）を書き出して新セッションへ切る判断基準を明文化
- 配置先: `.claude/rules/apd/01-phases.md` または新規 `07-context-management.md`
- 規定すべき内容:
  - リセット判定のシグナル（コンテキスト消費率、応答品質劣化、長時間化のしきい値）
  - 引き継ぎファイルに含める項目（残タスク、Spec参照、これまでの判断、未解決の問い）
  - リセット後の再開手順

**A-2. 評価ループの収束基準**
- Peer Review / 専任Checkpointの最大反復回数、収束しない場合のエスカレーション条件
- 配置先: `.claude/rules/apd/01-phases.md`
- 規定すべき内容:
  - 最大反復回数（例: 5回）
  - 収束しない場合のHuman Checkpointへのエスカレーション
  - 各反復で記録すべき指摘・修正サマリー

### B. 領域依存（テンプレート提供にとどめる）

**B-1. rubricベースの評価**
- UI/プロダクトデザインを伴うサイクル向けに、Spec側で評価軸を定義 → 専任Checkpointが多軸スコアで判定する拡張
- opt-in方式。汎用フレームワークの単純さは壊さない
- 配置先: `templates/` に rubric付きSpecテンプレート

**B-2. 動作検証エージェント**
- Playwright/CLIで実成果物を触るEvaluatorパターン
- 配置先: `templates/agents/` または `examples/`
- 任意導入。プロジェクトのCLAUDE.mdでオン/オフ

### C. 検討中

**C-1. ハーネスのコスト/時間制約**
- Build Phaseのタイムボックス、費用上限を設定する仕組み
- 必要性は高いが、Claude Code側のメトリクスとの連携が必要。設計にひと工夫いる
- 一旦保留、必要性が顕在化したら再検討

**C-2. ハーネスの陳腐化マネジメント**
- モデル進化に応じてハーネスを簡素化する考え方をフレームワーク自身に組み込む
- 取り込みはAPD自体のメタな話なので、ドキュメント（CHANGELOG.md または APD-FRAMEWORK.mdの末尾）で「ハーネス再評価のリズム」を明記する程度で十分か

## 議論の結果（最終決定）

### A-1: コンテキストリセット

- **デフォルト**: リセットしない。compactionに任せる
- **長時間Build時**: リーダーエージェントが**1コンテキストの実装+テスト完了境界**でリセット。途中の強制リセットはしない
- **判断主体**: リーダーエージェントの定性判断（応答品質劣化・残タスク量・消費量を踏まえる）
- **handoff document**:
  - リセット直前にリーダーが書く
  - 中身は state + next steps が最低限。具体構造はテンプレート提示にとどめる
  - Markdown + YAML frontmatter
  - 追記のみ（H-001, H-002 と積み上げる。Amendment形式は採らない）
  - 配置先: `docs/apd/cycles/C-{NNN}/handoffs/H-{NNN}.md`
  - 新セッションは最新1本を起点に再開

#### A-1の議論の経緯

私は当初「コンテキスト消費率70%」のしきい値ベース発火を提案したが、文献にない決め打ちだったため撤回。文献確認の結果「スプリント=作業単位完了境界が文献の素直な推奨」と判明し、APDの既存「コンテキスト並列実行」を作業単位として流用する形に落ち着いた。**「文献にないことは決めすぎない」を以降の原則にした。**

### A-2: 評価ループの収束基準

- **収束**: Specに定義された評価軸を満たし、リーダーエージェントが「これ以上の改善余地が乏しい」と判定したとき
- **行き詰まり時のトリアージ**: 同一論点で行き詰まりが続いた場合、Build起因 / Spec起因 を判別し、Spec起因なら Human Checkpoint へ
- **規定しないもの**: 反復回数の上限、プラトー判定アルゴリズム
- 評価軸の具体構造（binary/graded、スコア体系など）は **フレームワーク側で規定しない**。Spec が決める

#### A-2の議論の経緯

複雑化を何度も繰り返した。最初は「同一論点で連続3回 request_changes でエスカレーション」と数字を入れたが、ユーザーの「2値判定にこだわらない」という発言を受けて rubric 化を検討。「optional だと AI がサボる」という指摘でハイブリッド案を撤回し強制rubric化を提案、binary/graded type 分け・5段階スコア固定・Phase 1 Exit Criteria 追加など細かいルールを積み上げた結果、ユーザーから「考えすぎでは」と指摘され、**Specに評価軸を書く + リーダーの定性判断 + Build起因/Spec起因トリアージ** だけに収束した。

### B-1: rubric ベースの評価

A-2 と B-2 の「Specに評価軸（rubric）を書く」設計に統合された。独立した規定としてはクローズ。

### B-2: 動作検証と3点突合

- Specの各評価軸に **`verification`**（method + evidence_required + 必要なら note）を記述する
- Evaluator（Peer Review / 専任Checkpoint）は**実成果物を動作させて検証する**。コード読みだけでは不可
- 検証時に **evidence をファイル保存**（`docs/apd/cycles/C-{NNN}/evidence/{review-id}/{axis-id}/`）
- レビューサマリーに「軸ID → evidenceパス」の対応を含める
- **3点突合**: リーダーは収束判定前に **評価軸 × evidence × 実toolcall記録** の3点を突合。揃わない軸は差し戻す
- **実装エージェントが自分の実装を検証することは禁止**
- 自動検証不可な軸は `verification.note` で代替手段を明記
- 3点突合の具体実装手段（toolcall記録の取得方法等）は CLAUDE.md でプロジェクトごとに定義。フレームワークは責務のみ規定
- **Human Checkpoint は3つのまま**。HC2 の責務は「完成品確認」のみで、evidence確認は求めない

#### B-2の議論の経緯

「動作検証してないのに『動作検証しました』とAIが嘘をつく」問題への対応。当初「HC2でランダムサンプリング」を提案したが、APDの基本思想（人間負担を増やさない）に逆行することをユーザーから指摘され撤回。AI Checkpointで完結させる**3点突合**を主案に格上げし、人間の責務は据え置いた。

### C-1: ハーネスのコスト/時間制約

**規定しない**。文献はコスト/時間の結果報告であって、ハーネスに上限を組み込めという話ではない。Claude Code側のメトリクスとの連携が必要で抽象化しづらい。実運用で必要性が顕在化したら再検討する。

### C-2: ハーネスの陳腐化マネジメント

**APD-FRAMEWORK.md 末尾に「ハーネスの再評価」節を追加**。新規ルールではなく、設計姿勢の表明にとどめる。モデル進化に応じてフレームワーク自体を簡素化することを明記。

## メタな学び（議論プロセスから）

このサイクルで繰り返し起きた失敗パターン:

1. **文献にないことを決めてしまう** — A-1 でしきい値、A-2 で反復回数固定値・スコア体系固定など
2. **AIサボり防止のために構造を盛り込みすぎる** — A-2 で binary/graded type 分けや Phase 1 Exit Criteria 追加など。ユーザーから「考えすぎ」指摘で簡素化
3. **Human Checkpointを増やす方向に流れる** — B-2 で HC2 にスポットチェックを追加しようとして基本原則と矛盾

**ガイドライン:** 取り込み判断時は常に「文献にあるか / なくても本当に必要か / Human Checkpointを増やしていないか」を確認する。文献にないことを決めるときは、その決定の根拠を明示する。
