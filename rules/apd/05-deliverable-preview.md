# APD 成果物プレビュー

## 概要

実装前に「何ができるか」を可視化し、方向性のズレを早期に検出する。

- **Spec フェーズ**: テキストで完成像の方向性を合意し、生成すべきプレビュー種別を定義する
- **Build フェーズ**: 実装開始前に具体的な図・モック・ダイアグラムを生成する

プレビューは原則必須。最低 1 つ（アーキテクチャ図）はどのプロジェクトでも生成することを推奨する。

## プレビュー対象（最低 1 つ推奨、プロジェクトに応じて追加選択）

| 種別 | フォーマット | 例 |
|------|------------|-----|
| アーキテクチャ図 | Mermaid | システム構成、コンポーネント関係 |
| データモデル | Mermaid (ER 図) | テーブル設計、リレーション |
| シーケンス図 | Mermaid | API 呼び出しフロー、処理順序 |
| 画面遷移図 | Mermaid (stateDiagram) | ページ間のナビゲーション |
| UI モック | HTML | 静的 HTML+CSS による画面モック |
| API 仕様 | Markdown | エンドポイント、リクエスト/レスポンス例 |

## 成果物の配置

```
docs/apd/preview-{slug}/
├── architecture.md         ← Mermaid 図を含む Markdown
├── data-model.md
├── sequence-{name}.md
├── screens/
│   └── {screen-name}.html  ← HTML モック
└── api-spec.md
```

プレビューを 1 つしか作らない場合は `docs/apd/preview-{slug}.md` 単一ファイルでもよい。

## Spec フェーズでの方向性合意

Spec の `Deliverable Previews` セクションで、生成すべきプレビュー種別と方向性を記述する。「どんな画面があるか」「どんな API か」をテキストで合意する。

## Build フェーズでのプレビュー生成

Build フェーズの実装開始前に、Spec の記述に基づいてプレビューを生成する。プレビューはサイクル単位で管理する（イミュータブル、修正は Patch）。

## Acceptance でのレビュー

人間は PR 本文の「試し方」と合わせて、必要に応じてプレビューを目視確認する。プレビューが期待と大きく外れていれば差し戻し、Spec Patch サイクルで方向修正する。
