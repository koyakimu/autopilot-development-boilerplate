# APDフェーズ定義

## フェーズ

```
Phase 0: Design ── 人間 + AI 対話（並列化しない）
  成果物: プロジェクトデザイン文書（北極星）
  ─────────────── Human Checkpoint 0 ───────────────

Phase 1: Spec ── AIドラフト + 人間フィードバック（並列化しない）
  成果物: Spec集（AC + 評価軸rubric + verification + テスト戦略 + 成果物プレビュー記述）+ Decision Records（技術選定含む）
  ─────────────── Human Checkpoint 1 ───────────────
  ここから先、人間は意図を決めない。完成品を確認するだけ

Phase 2: Build ── AI自律・並列実行
  成果物: 成果物プレビュー + 実装 + テスト全パス + rubric evidence
  Peer Review + AI Checkpoint → Human Checkpoint 2（完成品確認）
```

## Checkpointの原則

- **Human Checkpoint**: 人間が意図を決める場（Phase 0-1）と、完成品を確認する場（Phase 2）に置く
- **AI Checkpoint**: 品質を機械的に検証する。Build Phase内で実装完了後に実行
- 人間はコードを読むことを求められない。完成品が意図通りに動くかを確認する
- エスカレーションが必要な場合のみ、人間に判断を仰ぐ

## AI Checkpointエスカレーションポリシー（デフォルト）

**Human Checkpoint必須:**
- 新しいビジネスルール（既存Specにないドメインロジック）
- 外部システムとのインターフェース変更
- セキュリティ・認証に関わる変更
- データモデルの破壊的変更
- パフォーマンス要件の緩和

**AI Checkpoint完結:**
- UI調整（Design文書の範囲内）
- 既存ビジネスルール内のバリエーション追加
- リファクタリング（振る舞い変更なし）
- テストカバレッジ補強
- ドキュメント文言修正

## Build Phaseの長時間化対応（コンテキストリセット）

長時間Buildでリーダーエージェントが作業単位の完了境界でコンテキストをリセットする運用。**デフォルトはリセットしない**（compactionに任せる）。

- **リセット境界**: 1コンテキストの実装+テスト完了など、中断不可な作業単位の完了境界。途中での強制リセットはしない
- **判断アンカー**: リーダーは以下のシグナルを踏まえて発火を定性判断する
  - 応答品質の劣化（同じ指摘の繰り返し、文脈喪失の兆候）
  - 残タスク量と現在のコンテキスト消費量のバランス
  - 直近のツール呼び出しでエラーの再発率が上がっているか
- **handoff document**: リセット直前にリーダーが書く。新セッションは最新handoffを起点に再開する（フォーマットは `03-documents.md` 参照）

## 評価ループの収束基準

Build Phaseの評価は、Specに定義された評価軸（rubric）に沿ってPeer Review / 専任Checkpointが反復する。

- **収束**: Specの評価軸が満たされ、リーダーが「改善余地が乏しい」と判定したとき
- **判断アンカー**: リーダーは収束判定で以下を見る
  - 直近Nラウンドで指摘事項のクリティカル度が下がっているか
  - 同じ論点が再出していないか
  - evidenceが全軸で揃っているか（後述の3点突合）
- **行き詰まり時のトリアージ**: 同一論点で詰まったら、`02-cycle-flow.md` の **バグトリアージ（Build起因/Spec起因）と同じ判定軸**を適用する。Spec起因なら Human Checkpoint へ、Build起因なら通常ループ継続
- 反復回数の上限・プラトー判定アルゴリズムは規定しない

## 動作検証とevidence

**テストはACのカバーを保証し、動作検証はrubric軸（テスト化困難な品質：UI、E2E、パフォーマンス、AI出力品質等）を保証する。** 両者は別レイヤで併存する。

Evaluator（Peer Review / 専任Checkpoint）は、Specの各評価軸について**実成果物を動作させて検証する**。コード読みだけでは収束させない。

- 検証手段は Spec の `verification.method` で指定（Playwright MCP、curl、テストランナー、CLI実行等）
- evidence をファイル保存（配置先は `03-documents.md` 参照）
- レビューサマリーに「軸ID → evidenceパス」を必ず含める
- 自動検証が原理的に不可能な軸は Spec の `verification.note` で代替手段を明記
- **実装エージェントが自分の実装を検証することは禁止**。Peer Review / 専任Checkpoint が行う

### 3点突合

リーダーは **最終収束判定時** に **評価軸 × evidence × 実toolcall記録** の3点を突合する。揃わない軸は差し戻す。各反復ラウンドでの突合は不要（evidence は反復ごとに更新されるため）。

- 3点突合の実装手段（toolcall記録の取得方法、突合スクリプト等）はプロジェクトのCLAUDE.mdで定義する
- **フォールバック**: CLAUDE.mdで toolcall記録の取得手段が未定義の場合、リーダーは **2点突合（評価軸 × evidence）** に縮退してよい。この場合、evidenceの内容妥当性チェックを通常より厳しめに行う
- フレームワークは責務のみを規定し、具体実装は環境依存

### 並列実行（worktree）との両立

`06-git-strategy.md` のworktree並列実行を採用している場合、検証主体（Peer Review / 専任Checkpoint）が別worktreeから実成果物に到達できないことがある。このため動作検証は2段階に分離する:

- **コンテキスト単位の動作検証**（worktree内で完結する範囲）: ユニット範囲のCLI動作確認・curl等は **Peer Review（別コンテキスト担当のレビューエージェント）** が同一worktree内で実施
- **結合動作検証**（worktree横断・E2E相当）: 全worktreeをマージしたmainライン or 統合branchで、**専任Checkpointエージェント**が一括実施。`apd-framework.md` のBuildフロー mermaidの「結合検証」ノードがこのフェーズに該当

worktree内では実装エージェントは**ビルド可能状態とユニットテスト全パスまで**を責務とし、コンテキスト単位の動作検証は実装エージェントと別のレビューエージェントが行う。「実装した本人が動作検証する」ことは禁止という原則は両段階で保たれる。
