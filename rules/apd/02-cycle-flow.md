# APD サイクル型統一フロー

## サイクル型統一フロー

すべての変更を「サイクル」として統一する。

| トリガー | フロー |
|---------|--------|
| 新プロダクト / 大きな方向転換 | Intent (Design) → Spec → Build → 完成後の実機確認 |
| 新機能追加 | Spec（既存 Design 参照、新規 spec ファイル）→ Build → 完成後の実機確認 |
| バグ修正 / 仕様変更 | 既存 Spec を編集（version を上げる）→ Build → 完成後の実機確認 |
| 技術的変更（リファクタ等）| `decisions.md` に追記 → Build → 完成後の実機確認 |

GitHub issue がある環境では **1 issue = 1 サイクル** を基本とする。issue 番号と Spec / PR を相互リンクすることでサイクルの trail が自動で残る。issue がない環境では `docs/apd/` のファイルと git 履歴でサイクルを表現する。

## バグのトリアージ

バグ報告・テスト失敗が来たとき、AI がまず原因を判定する:

- **Spec 起因**（仕様漏れ・曖昧さ）→ 完成後の実機確認で気づき、次サイクルで Spec を修正して version を上げる
- **Build 起因**（実装が Spec と合っていない）→ 実装中に Spec チェック Stop フックが検出し、AI が自律修正する
